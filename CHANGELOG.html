## ⚠️ 巨坑备忘：日次日期公式 vs 取引行插入

### 现象
取引里明明有今天的数据，日次的 A 列就是不显示今天 —— 但今天之前的日期合计（B 列）却正常更新了。

### 根本原因
Google Sheets 有一个「自动修正引用」行为：**当你在某个范围的起始行之前插入新行时，Sheets 会自动把公式里的引用往下顺延。**

例如：
- 日次 A2 原本写的是 `FILTER('取引'!A2:A, ...)`
- 你在取引第 2 行插入一行（顶部追加新交易）
- Sheets 自动把日次 A2 的公式改成 `FILTER('取引'!A3:A, ...)`
- 从此新插入的顶部行永远在 A3 之前，永远被跳过
- **这个修改是静默的**，公式栏里看起来「正常」，但范围已经偏了

### 为什么 B 列合计不受影响？
B2 的 SUMIF 用的是 `取引!A:A`（全列），不存在起始行，所以插入行不会触发引用偏移，金额照常统计。这也是为什么症状很迷惑：旧日期的钱对了，但新日期根本不出现。

### 修复方案
日次 A2 公式改用 `INDIRECT` 把范围写成字符串，Sheets 就无法自动修改它：

```
=IFERROR(SORT(UNIQUE(FILTER(INDIRECT("'取引'!A2:A"), INDIRECT("'取引'!A2:A")<>"")), 1, FALSE), "")
```

`INDIRECT("'取引'!A2:A")` 是一个文本求值，Sheets 的自动引用修正机制对它无效，范围永久锁死在 A2。

### 操作规范
- 取引的新数据可以**顶部插入**，日次公式不再受影响
- 每次跑 `setupDailySheet()` 会重置为正确的 INDIRECT 公式
- **不要手动编辑日次 A2 的公式**，否则可能无意间去掉 INDIRECT 又踩坑

## 调试日志说明

运行任意函数后，在 Apps Script → 执行记录 中可看到以下 log：

| 前缀 | 含义 |
|---|---|
| `[parser] card_notification` | 解析到刷卡通知邮件，显示商户名 |
| `[parser] bank_withdrawal` | 解析到银行出金邮件 |
| `[parser] pre_debit_notice` | 解析到口座引落事前通知，显示件数和商户列表 |
| `[parser] WARN: 全パーサー不一致` | 邮件格式未匹配任何解析器，需检查新格式 |
| `[generateSummary_]` | 统计期间、计入金额、各类跳过行数、oldCol_fallback 数 |
| `[generateSummary_] WARN: 列缺失` | 取引 sheet 列结构异常，dateCol 或 amtCol 找不到 |

### 排查金额不符时的步骤

1. 运行 `sendCurrentMonthExpenseReport()`
2. 查看执行日志中 `[generateSummary_]` 那行
3. 若 `skip(no_cat=N)` 较大 → 有 N 行无分类被跳过（通常是旧格式数据）
4. 若 `oldCol_fallback=N` > 0 → N 行使用了旧「分類」(O) 列，说明迁移数据存在
